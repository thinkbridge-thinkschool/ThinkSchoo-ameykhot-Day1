thinkschool-sql-amey.database.windows.net  |  thinkschool-quotesdb-amey  |  Isolation Level Demo — ANOMALY 2: NON-REPEATABLE READ
⚠️  Non-Repeatable Read: Session 2 reads the same row twice within one transaction and gets different values — because Session 1 committed an UPDATE in between.
SESSION 2  READ COMMITTED — reads same row twice, gets two different values
SET TRANSACTION ISOLATION LEVEL READ COMMITTED; BEGIN TRANSACTION; -- First read SELECT Id, Text FROM Quotes WHERE Id = 1; WAITFOR DELAY '00:00:05'; -- window for S1 to UPDATE -- Second read (same query) SELECT Id, Text FROM Quotes WHERE Id = 1; COMMIT;
▶ Result — NON-REPEATABLE READ OBSERVED
S2 — First read (t=0):
IdText
1You have power over your mind not outside events
(1 rows affected)
<< waiting 5 seconds... Session 1 updates in between >>

S2 — Second read (t=5, same query):
IdText
1UPDATED BY SESSION 1
(1 rows affected)
⚠ Same query, same transaction — two different values!
SESSION 1  Commits UPDATE while Session 2 is between reads
-- Runs at t+2s while Session 2 is in WAITFOR UPDATE Quotes SET Text = 'UPDATED BY SESSION 1' WHERE Id = 1; COMMIT; -- committed before S2's second read
▶ Output
(1 rows affected)
S1: Row updated and committed at t+2s

Session 1 committed its change between Session 2's first and second reads.
Under READ COMMITTED, Session 2's second read picks up the new committed value — hence the anomaly.
